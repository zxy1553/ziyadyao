## 3. 日常开发困扰 (P3: 痛点分析)
**(动作：语气稍作停顿，以此引起听众对问题的共鸣)**

在深入业务一线的过程中，我并没有止步于完成任务，而是敏锐地发现了一些制约我们开发效率的**“三大痛点”**：

1.  **重复造轮子**：在战斗基建中，我们有大量结构相似的代码（如上百个AbleTask）。目前的脚本工具只能生成空框架，具体的逻辑内容仍需我们手动一行行填充，重复劳动极多。
2.  **API检索低效**：我们的开发文档分散在企微、iWiki和代码注释中。开发时为了找一个API，往往需要在多个平台间反复横跳，且文档维护往往滞后于代码迭代。
3.  **Debug流程繁琐**：特别是在QA验收阶段，排查DS（Dedicated Server）问题时，我们需要手动拉取日志、人工过滤、肉眼比对源码。这个过程机械且耗时，严重挤占了分析问题本身的时间。

面对这些困扰，传统的人力堆叠模式显然效率难以提高。因此，我开始思考：**如何利用AI Agent技术来打破这一僵局？**

## 4. 解决方案：Skill 实现自动化 Dev (P4-P7)
**(动作：引导视线对比左右两边的流程)**

为此，我引入了**“Skill”**机制来实现自动化开发。

### 案例一：Status Writer Skill
让我们以**Status Writer Skill**为例。
*   **右边是日常流程**：以前要实现一个Status，我需要先阅读大量文档，理解基类、钩子函数，然后小心翼翼地编写代码。这要求开发者对框架有很深的理解。
*   **左边是Skill流程**：现在，我只需要告诉AI Agent一句话：“我要实现一个Status管理角色武器mesh的显隐”。
    *   Agent就能自动理解需求，设计好GameplayTag；
    *   自动生成包含完整逻辑的Lua代码；
    *   并列出所有需要在蓝图和DataTable中配置的条目。
我只需要做简单的配置和测试，**三步即可轻松实现**。这不仅将开发门槛降到了最低，更保证了代码规范的一致性。

### 案例二：AbleTask Skill
**(动作：强调“一步完成”)**
同样的思路，我也应用到了**AbleTask**的开发中。
*   **传统模式下**：脚本只能帮我们生成一个空壳，具体的API调用逻辑（比如如何获取Owner、如何操作黑板）全靠人工检索和编写。
*   **使用AbleTask Skill后**：AI不仅能生成框架，更能根据我的自然语言描述（例如“通过Status在Task期间改变角色武器显隐”），自动检索最佳实践的API，并**自动填充具体逻辑**。
它真正实现了从“填空题”到“全自动生成”的跨越，让开发者可以专注于业务逻辑的设计，而不是代码实现的细节。

## 5. 解决方案：Skill 实现自动化 Debug (P8-P9)
**(动作：轻松的语气)**

### 案例三：DS Log Dump Skill
针对繁琐的Debug流程，我开发了**DS Log Dump Skill**。
以前，为了拿一份DS日志，我们需要登录后台、输入BattleID、等待下载，过程非常繁琐。
现在，我只需要在对话框里发一句指令：“帮我获取这个BattleID的日志”。AI就会自动执行脚本，将日志拉取并发送给我。
这看似是一个小小的改动，但它极大地缩短了Debug的反馈回路。

### 案例四：Log Split Skill
**(动作：承接上一页，语气转折)**
虽然实现了日志自动拉取，但在实际落地中，我遇到了一个新的挑战：**日志文件太大了**。
动辄上百兆的DS日志，如果直接丢给AI模型，不仅会瞬间撑爆Token限制，还会导致关键信息被淹没。为此，我进一步开发了**Log Split Skill**。
它就像一个智能的“预处理器”，能够自动识别日志结构，剔除无用的噪音数据，并将大文件精准切分为AI可阅读的小片段。这解决了自动化Debug的“最后一公里”问题，确保Agent不仅能“拿到”数据，还能真正“看懂”数据。

## 6. 核心技术：知识库与 RAG (P10-P14)
**(动作：展示思考的深度，手势指向三个Case)**

### 知识库设计：Why & How
在有了工具（Skill）之后，我发现Agent依然会犯错。经过复盘，我总结了三大原因：
1.  **幻觉问题**：Agent可能会编造不存在的API。
2.  **归因错误**：比如在Debug时，Agent因为不懂我们的Status刷新机制，错误地将正常逻辑判定为Bug。
3.  **效率低下**：检索信息依然很慢。

为了解决这些问题，我引入了**RAG（检索增强生成）技术**，构建了一个分层的知识库架构。
*   **底层**是基础的开发与Debug知识；
*   **上层**是特定领域的业务逻辑（Domain Knowledge）。
这相当于给Agent装上了一个“外挂大脑”，让它从一个“懂代码的实习生”进化为一个“懂项目的资深开发”。

### Dev 工作流对比
**(动作：对比左右两图，强调结果)**
让我们通过实战来看看效果。
*   **左边是未使用RAG的情况**：Agent在代码库中漫无目的地“大海捞针”，它找到的参考代码往往是过时的或者错误的（比如这里它错误地使用了ScratchPad），导致生成的代码无法运行。
*   **右边是Skill+RAG的工作流**：Agent首先通过RAG检索知识库，直接锁定了最佳实践模板。它清楚地知道应该使用`GetAbilitySystemComponent`而不是错误的API。
结果就是：**一次生成，直接编译通过**。这不仅提升了速度，更重要的是保证了代码的**准确性**和**鲁棒性**。

### Debug 工作流对比
**(动作：这是最精彩的案例，语速放缓，强调Debug Agent的优势)**
在Debug场景下，这种优势更为明显。我们以“星宝使用相机后道具消失”这个棘手的Bug为例：
*   **普通的AI**（如TAPD AI）只能给出“模型加载失败”这种泛泛而谈的猜测。
*   **Default Agent**虽然能看日志，但因为它不懂业务，它看到Status被移除就以为是问题所在，实际上那是正常的刷新逻辑。
*   **而我的Debug Agent**（最右侧）：因为它结合了**日志信息**和**项目背景知识**，它精准地识别出Status刷新是正常的，并通过对比Tag计数，敏锐地发现`HideWeapon` Tag的计数在移除时没有归零。
它直接指出了**根本原因**，并给出了修复建议。这就是“懂业务”的AI与普通AI的本质区别。

## 7. 系统架构总结 (P15)
**(动作：展示大局观)**

最后，我将上述所有工作整合成了一套**模块化的战斗Agent系统**。
*   **顶层**是主Agent进行任务路由；
*   **中间层**是专精的SubAgent，分别负责开发和Debug；
*   **底层**则是由记忆系统、Skill库和MCP工具组成的坚实基座。

这不仅仅是一些脚本的堆砌，而是一套**可扩展、可复用、低耦合**的工业级解决方案。
